# ActionRPG

`ActionRPG` 是一个基于 Unreal Engine 5.4 的单人动作游戏战斗框架工程。

这个仓库的目标不是演示某一个独立技能、某一条动画链，或一段临时玩法原型，而是沉淀一套可以继续扩展的动作战斗主链：把角色、武器、输入、命中、受击、切武、处决和 Spirit 技能收进统一运行时语义里，方便后续继续接玩法、反馈、UI 和状态效果。

如果你正在寻找一套可以继续往下做武器玩法、命中反馈、能力系统和战斗表现扩展的 Unreal 动作游戏基础工程，这个仓库就是围绕这个目标组织的。

> [!IMPORTANT]
> 仓库包含 `.uasset` / `.umap` 等 Unreal 大文件资源。完整克隆并正常打开工程前，请先安装 `Git LFS`，然后执行 `git lfs install` 和 `git lfs pull`。

## 项目定位

当前仓库聚焦的是一套正式战斗骨架，而不是一次性演示内容。

它重点解决 4 件事：

1. 稳定角色战斗主链，避免攻击、受击、切武、处决之间长期互相污染。
2. 把武器、攻击段、命中配置和动画差异尽量收口到数据资产与公共类型层。
3. 把资源加载、装备状态和战斗运行态分层，降低后续扩展成本。
4. 在不依赖大量蓝图硬编码的前提下，保留继续接 UI、反馈和高级玩法的空间。

## 你会在这个仓库里看到什么

这个项目不是把 `Attack`、`WeaponSwitch`、`Execution`、`Spirit` 这些功能分散堆在角色蓝图里各自处理，而是把它们放进同一套战斗框架里统一驱动。

更具体地说：

1. 输入不会直接跳过战斗系统落到动画，而是先进入统一战斗入口做资格判断和状态推进。
2. 武器差异不会主要靠分叉蓝图硬编码，而是尽量通过 `WeaponDefinition`、攻击段配置和公共运行时组件来表达。
3. 命中、受击、破韧、处决等结果不会各自维护一套平行状态，而是尽量回到统一的结果链和运行时状态源。
4. 动画和通知负责表现与精确时机，不负责充当正式状态机，这样后续继续扩玩法时更容易收口。

## 当前已落地的核心能力

### 固定四槽武器体系

当前角色战斗围绕固定四槽组织：

1. 空手槽
2. 近战槽
3. 远程槽
4. 近远程混合槽

这套结构服务的是动作游戏里的稳定输入语义、武器切换和攻击路由，而不是通用背包系统。它的价值在于：角色始终围绕同一套装备入口和切换语义工作，后续接更多武器类型时不需要先重做整套装备主链。

### 主动战斗主链

当前已经正式接入并持续维护的主动链包括：

1. Attack
2. Defense
3. Dodge
4. WeaponSwitch
5. Execution
6. Spirit

这些链路统一通过角色组件、Gameplay Ability、武器定义和命中解析器协作推进，而不是散落在角色蓝图里各自维护状态。这样做的意义是：当你后续继续做新武器、加新段数、补新命中表现时，可以沿用已有主链，而不是每条功能线都重开一套状态管理。

### 命中、受击与结算基础链

当前仓库已经具备完整的基础战斗事件流：

1. 攻击请求解析
2. 命中窗口开启
3. 命中载荷构建
4. 命中解析与伤害结算
5. 受击结果广播
6. 受击表现与恢复阶段推进

这让普通命中、格挡、破韧、处决命中和后续扩展效果都能围绕统一结果链工作。对二次开发来说，这意味着新效果更容易接在正式落点上，而不是靠临时事件拼接。

### 装备预热与运行时资源桥

仓库当前不把“正式装备态”和“资源是否 ready”混成一层。

已经落地的运行时能力包括：

1. 武器定义异步加载
2. 启动期预热
3. 固定槽位武器实例生成与复用
4. 当前武器快照同步
5. 切武后的动画层与挂点切换

这为后续减少战斗内切换抖动、接装备 UI 和加载态说明提供了正式入口，也让“角色当前装备了什么”和“相关资源有没有准备好”可以被分层处理。

## 高层架构理解

如果你第一次阅读这个工程，可以先按下面这套分工理解它：

### 角色组件

角色组件负责维护正式运行态，例如当前武器、战斗模式、输入消费、命中窗口、装备快照和部分公共战斗状态。它们更像“主链状态源”和“运行时协调层”。

### Gameplay Ability

Gameplay Ability 负责承接主动行为的起手、资格判断、表现推进和收尾。例如攻击、切武、处决、Spirit 等行为会通过 Ability 进入正式链路，而不是直接由输入去驱动动画。

### WeaponDefinition 与配置资产

武器配置负责表达武器差异，包括武器类别、攻击段、命中配置、Spirit 配置和部分表现差异。它承担的是“数据驱动入口”，让新增武器玩法时优先改配置和少量公共逻辑，而不是复制整套功能蓝图。

### 动画与通知

动画、蒙太奇和通知负责表现、窗口时机和段落切换。它们很重要，但默认不承担“正式业务状态源”的职责。这样做的好处是：即使动画层和玩法层同时变动，核心状态也仍然能收在代码与正式运行时组件里。

## 适合拿来做什么

这个仓库当前最适合被当成下面三类基础工程：

### 1. 动作游戏战斗框架模板

如果目标是尽快拥有一套可继续扩展的动作战斗骨架，这里已经提供了：

1. 角色战斗装配链

### 2. 数据驱动武器与能力系统原型

如果目标偏系统验证，当前仓库已经覆盖：

## 仓库结构

### `Source/`

正式代码入口，包含角色装配、战斗组件、Ability、命中解析、装备运行时、公共类型和数据资产接口。理解主链逻辑时，这里是最核心的阅读入口。

### `Config/`

工程配置、GameplayTag 配置和运行时参数入口。很多正式语义会先在这里定义，再被代码和资产共同消费。

### `Content/Player/`

玩家角色、武器、动画、能力蓝图和默认战斗资产。这里聚合了当前玩家侧主链所需的大部分正式内容。

### `Content/GameBase/`

项目基础运行壳，包括 GameMode、PlayerController、HUD 和相关运行蓝图。它负责把底层战斗能力接到实际可运行的游戏壳中。

### `Content/Enemy/`

当前敌人测试蓝图与敌人表现入口，用于承接受击、处决目标、基础交互和战斗验证。

### `Content/UI/` 与 `Content/Map/`

正式主链使用的 UI 与地图内容，用来支撑工程可运行、可测试、可演示的基本体验。

## 如何使用

### 环境前置

1. 安装 Unreal Engine 5.4
2. 安装 `Git LFS`
3. 克隆仓库后，在仓库根目录执行：

```bash
git lfs install
git lfs pull
```

### 打开工程

使用仓库根目录下的 `ActionRPG.uproject` 打开工程。

如果你是在本机首次打开 Unreal C++ 工程，按你的本机环境正常生成项目文件即可。通常情况下，先确认 LFS 资源已完整拉取，再让 Unreal Editor 完成首次工程加载，会比一开始就急着排查资源缺失更稳妥。

### 本地构建

如果需要在本机从源码验证工程状态，请使用你自己安装的 Unreal Engine 5.4 的 `Build.bat`，并把下面的占位路径替换成你的实际目录：

```powershell
<UE_5.4_ROOT>\Engine\Build\BatchFiles\Build.bat ActionRPGEditor Win64 Development "<PROJECT_ROOT>\ActionRPG.uproject" -waitmutex
```

其中：

1. `UE_5.4_ROOT` 表示你本机的 Unreal Engine 5.4 安装目录
2. `PROJECT_ROOT` 表示你本地克隆下来的仓库目录

### 推荐的首次验证方式

如果你第一次接触这个工程，建议按下面顺序验证：

1. 先确认 `Git LFS` 资源已经完整拉取。
2. 使用 Unreal Editor 打开 `ActionRPG.uproject`，确认工程能正常进入编辑器。
3. 浏览 `Content/Player`、`Content/GameBase` 和 `Content/Enemy`，快速理解玩家、运行壳和敌人测试入口分别在哪里。
4. 如需验证 C++ 状态，再按上面的方式本机构建 `ActionRPGEditor Win64 Development`。

如果你只是想先体验当前工程内容，直接用 Unreal Editor 打开 `ActionRPG.uproject` 通常就够了。

## 当前公开仓库边界

这个公开仓库只保留当前正式主链内容，不把本地协作资料和第三方演示大包一起带上。

默认边界是：

1. 保留 `Source/`、`Config/`、`Content/Player`、`Content/GameBase`、`Content/Enemy`、`Content/UI`、`Content/Map` 等正式主链目录。
2. 排除本地开发规范、个人协作目录、缓存目录和临时输出目录。
3. 不把第三方角色包、演示大包和固定禁推资产当成公开仓库正式内容。

这样做的目的很直接：让公开仓库聚焦在当前真正持续维护的战斗主链和运行壳内容上，避免下载者把第三方包、个人资料和本地工具输出误以为是正式工程的一部分。

当前明确不作为公开正式内容的典型类别包括：

1. 本地 `Docs` 与协作规范目录
2. 第三方角色包和演示资源包
3. 临时缓存、工具输出和个人目录
4. 固定禁推的过渡 / 第三方大包资产

## 当前状态与后续方向

当前仓库已经具备一套清晰的动作战斗主链，但它仍然是“框架工程”，不是完整商业游戏成品。

已经具备的优势是：

1. 角色、装备、战斗、命中和受击职责分层已经比较清晰。
2. 武器差异正在稳定转向数据驱动。
3. 处决、切武、Spirit 和受击反馈已经有统一运行时落点。
4. 后续继续扩 UI、锁定、反馈和高级规则时，不需要从零重做主链。

它当前还不是什么：

1. 不是完整内容量的商业游戏样例。
2. 不是围绕单一武器或单一关卡包装好的成品 Demo。
3. 不是一份覆盖所有开发规范和项目流程的内部开发手册。

如果你打算继续二次开发，当前最自然的切入方向包括：

1. 扩更多武器类型和武器差异配置。
2. 继续完善 Spirit 技能、多段链和更多主动能力分支。
3. 补锁定目标、镜头反馈、伤害数字、特效、音效和 HUD 表达。
4. 在现有命中与受击链上继续扩状态效果、持续修正和更复杂的打断关系。

## 总结

`ActionRPG` 当前更像是一套已经把正式主链搭稳的 Unreal 动作战斗框架，而不是一个只演示少量孤立功能的样例项目。

如果你需要的是一套可以继续往下做武器玩法、命中反馈、UI、能力系统和高级战斗规则的基础工程，这个仓库已经把最重要的骨架部分准备好了。
